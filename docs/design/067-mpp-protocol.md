# Design: MPP Payment Scheme Support (Lightning Charge)

**Issue:** #67  
**Author:** Toshi  
**Date:** 2026-04-15  
**Status:** Proposed

## Problem

`bolt402` currently understands only L402 / LSAT wire formats:

- `WWW-Authenticate: L402 ...`
- `Authorization: L402 <macaroon>:<preimage>`

That keeps the SDK Lightning-native, but it cannot talk to APIs that have moved to the MPP `Payment` authentication scheme. MPP matters because it is becoming the general HTTP 402 standard, while still supporting Lightning as a first-class payment method.

The issue asks for MPP support without turning `bolt402` into a multi-rail wallet. That means one thing in practice:

- speak MPP on the wire
- pay only when MPP offers `method="lightning"`
- reject non-Lightning payment methods clearly

## Scope

### Phase 1, in scope

Add support for MPP over HTTP for:

- `WWW-Authenticate: Payment ...`
- `Authorization: Payment <base64url-json>`
- `Payment-Receipt: <base64url-json>`
- `method="lightning"`
- `intent="charge"`

### Explicitly out of scope for Phase 1

- MPP `intent="session"`
- non-Lightning payment methods (`tempo`, `stripe`, `solana`, etc.)
- MCP / WebSocket MPP transports
- body-streaming or top-up session flows

This is the right first cut because the current `L402Client` already models a one-shot request, pay, retry flow. Lightning `charge` maps onto that. Lightning `session` does not.

## Why phase 1 stops at Lightning charge

MPP Lightning `session` is not just a new header format. It changes the client model:

- open session with deposit invoice
- keep a bearer session alive across many requests
- handle top-up and close flows
- manage refunds via return invoice

That needs a distinct session API and different caching semantics. Trying to squeeze it into the current `L402Client::get/post` flow would be fake progress.

## Protocol comparison

### L402 today

Challenge:

```http
WWW-Authenticate: L402 macaroon="<base64>", invoice="<bolt11>"
```

Credential:

```http
Authorization: L402 <macaroon>:<preimage>
```

Receipt model today:

- local `Receipt` generated by the client after successful retry
- no standard server receipt header is parsed

### MPP Lightning charge

Challenge:

```http
WWW-Authenticate: Payment id="...", realm="...", method="lightning", intent="charge", request="<base64url-jcs-json>"
```

Decoded request object shape for Lightning charge:

```json
{
  "amount": "100",
  "currency": "sat",
  "description": "Premium API access",
  "methodDetails": {
    "invoice": "lnbc...",
    "paymentHash": "<hex>",
    "network": "mainnet"
  }
}
```

Credential:

```http
Authorization: Payment <base64url-json>
```

Decoded credential shape:

```json
{
  "challenge": {
    "id": "...",
    "realm": "api.example.com",
    "method": "lightning",
    "intent": "charge",
    "request": "<base64url-jcs-json>",
    "expires": "2026-04-15T12:05:00Z"
  },
  "payload": {
    "preimage": "<64-char lowercase hex>"
  }
}
```

Receipt:

```http
Payment-Receipt: <base64url-json>
```

Decoded receipt shape:

```json
{
  "method": "lightning",
  "challengeId": "...",
  "reference": "<payment-hash>",
  "status": "success",
  "timestamp": "2026-04-15T08:00:00Z"
}
```

## Key design decision

### Decision: keep one client, auto-detect the wire protocol

`L402Client` should remain the high-level entry point for paid HTTP requests. Internally it should detect whether a 402 response is:

- L402 / LSAT
- MPP `Payment` with Lightning `charge`

The payment engine stays Lightning-only either way.

Why this is the right fit:

1. The user intent is still the same: fetch a paid resource over HTTP.
2. `LnBackend` already abstracts invoice payment correctly.
3. Most of the flow is identical: request, 402, pay invoice, retry.
4. Forcing callers to choose “L402 mode” vs “MPP mode” upfront is unnecessary friction.

## Internal model

Introduce a protocol-neutral internal challenge type in `bolt402-core`:

```rust
enum HttpPaymentChallenge {
    L402(L402Challenge),
    MppLightningCharge(MppChallenge),
}
```

And a protocol-neutral retry credential type:

```rust
enum HttpPaymentCredential {
    L402(L402Token),
    Mpp(MppCredential),
}
```

This keeps the public API stable while avoiding L402-specific assumptions in the request pipeline.

## Parsing strategy

### `bolt402-proto`

Add MPP primitives alongside existing L402 primitives:

- `MppChallenge`
- `MppCredential`
- `MppReceipt`
- `MppChallengeRequest`
- `MppLightningChargeRequest`
- `MppError`

Responsibilities:

- parse `WWW-Authenticate: Payment ...`
- decode `request` as base64url JSON
- validate `method == "lightning"`
- validate `intent == "charge"`
- extract authoritative invoice from `methodDetails.invoice`
- validate optional convenience fields:
  - `paymentHash` matches invoice payment hash when present
  - `network` matches invoice prefix when present
- parse and format `Authorization: Payment ...`
- parse `Payment-Receipt`

### Header selection

MPP can return multiple `WWW-Authenticate` headers in one 402 response. The client must inspect all of them and select the first supported challenge.

Selection order:

1. any valid `L402` / `LSAT` challenge
2. any valid MPP `Payment` challenge with `method="lightning"` and `intent="charge"`
3. otherwise return a clear unsupported-method error

That gives bolt402 compatibility with both ecosystems without ambiguity.

## Credential and cache semantics

This is the biggest protocol difference.

### L402

Today `bolt402` caches `(macaroon, preimage)` per endpoint because L402 credentials are reusable for the protected resource until the server rejects them.

### MPP charge

MPP credentials are single-use. The spec explicitly treats them as valid for exactly one request.

That means:

- **do not reuse MPP credentials from the token store**
- **do not write MPP credentials into the existing `TokenStore`**
- keep `TokenStore` unchanged for Phase 1, but use it only for L402 flows

This is the cleanest way to add MPP without breaking the current storage trait or pretending MPP credentials behave like macaroons.

## Budget handling

`BudgetTracker` and `LnBackend` stay unchanged.

Flow for MPP Lightning charge:

1. parse MPP challenge
2. extract BOLT11 invoice from `methodDetails.invoice`
3. decode amount from the invoice
4. enforce budget before payment
5. pay via existing `LnBackend::pay_invoice(...)`
6. build `Authorization: Payment ...` credential using the returned preimage

No new budget model is needed.

## Request body digest binding

MPP adds optional `digest` on the challenge. This matters for `POST` requests.

Phase 1 behavior:

- if `digest` is absent, current retry behavior is fine
- if `digest` is present and the original request has a body, compute the body digest locally and verify it matches before paying
- if it does not match, return a client-side protocol error without paying

This prevents paying for a challenge bound to a different request body.

## Receipt handling

Current `Receipt` is a local audit record. We should keep that, but enrich it for MPP.

Phase 1:

- continue producing the existing local `Receipt`
- parse `Payment-Receipt` when present on MPP responses
- validate:
  - `method == "lightning"`
  - `challengeId` matches the paid MPP challenge
  - `reference == payment.payment_hash`
  - `status == "success"`
- store the parsed receipt in an optional field on the local receipt model, or on the HTTP response wrapper

This gives us interoperability without throwing away the SDK’s own audit record.

## Error handling

Add new client-facing errors in `bolt402-proto::ClientError`:

- `UnsupportedPaymentMethod { method: String }`
- `UnsupportedPaymentIntent { intent: String }`
- `NoSupportedPaymentOption`
- `InvalidReceipt { reason: String }`

Rules:

- if the server offers only non-Lightning MPP methods, return `NoSupportedPaymentOption` or `UnsupportedPaymentMethod`
- if the server offers Lightning but only `intent="session"`, return `UnsupportedPaymentIntent`
- if receipt verification fails after a successful retry, return `InvalidReceipt`
- keep existing L402 errors unchanged

## Client flow after the change

```text
request(url)
  -> if cached L402 token exists, try it first
  -> send unauthenticated request
  -> if status != 402, return response
  -> inspect all WWW-Authenticate headers
  -> choose supported challenge:
       - L402 / LSAT
       - or Payment + lightning + charge
  -> extract invoice
  -> enforce budget
  -> pay invoice via LnBackend
  -> build protocol-specific Authorization header
  -> retry request
  -> verify protocol-specific receipt semantics
  -> cache only when protocol == L402
  -> record local receipt
```

## WASM impact

Phase 1 remains WASM-compatible because:

- invoice payment still goes through the existing `LnBackend` abstraction
- MPP parsing is pure data work in `bolt402-proto`
- no new runtime assumptions are required

## Testing plan

### Unit tests, `bolt402-proto`

- parse valid `Payment` challenge with Lightning charge request
- reject non-Lightning method
- reject unsupported intent `session`
- reject mismatched `paymentHash`
- reject mismatched `network`
- build and parse `Authorization: Payment ...` credential
- parse `Payment-Receipt`

### Unit tests, `bolt402-core`

- choose supported MPP challenge from multiple `WWW-Authenticate` values
- prefer supported challenge over unsupported payment methods
- do not write MPP credentials into `TokenStore`
- verify `Payment-Receipt` against challenge id and payment hash
- reject digest mismatch before payment

### Integration tests, `bolt402-mock`

Add MPP mode to the mock server:

- return `WWW-Authenticate: Payment ...` with Lightning charge request
- verify `Authorization: Payment ...` preimage credential
- return `Payment-Receipt`

Integration cases:

- MPP Lightning charge happy path
- MPP challenge without Lightning option
- MPP challenge with only `intent="session"`
- repeated request re-pays in MPP mode because credentials are single-use

## Implementation plan

1. Add MPP protocol types to `bolt402-proto`
2. Add new client errors for unsupported payment options and invalid receipts
3. Teach `bolt402-core` to select and execute MPP Lightning charge flows
4. Extend `bolt402-mock` with MPP challenge / credential / receipt support
5. Add unit and integration tests
6. Update README and tutorials once the implementation lands

## Consequences

### Good

- `bolt402` stays Lightning-maximalist while becoming MPP-compatible
- no changes needed to `LnBackend`
- no fake multi-rail support
- no premature session API
- no forced public API split between L402 and MPP callers

### Tradeoff

MPP `session` remains unsupported after Phase 1. That is intentional. It should become a separate design and implementation track rather than leaking session complexity into a request-retry client built for charge flows.
