# Design: CLN (Core Lightning) Backend Adapter

**Issue:** #52
**Author:** Toshi
**Date:** 2026-03-21
**Status:** Implementing

## Problem

bolt402 currently supports LND, NWC, and SwissKnife as Lightning backends. Core Lightning (CLN) is the second most widely deployed Lightning implementation, maintained by Blockstream. Users running CLN nodes cannot use bolt402 without switching implementations or using an intermediary like NWC.

## Proposed Design

### New Crate: `bolt402-cln`

Follows the same adapter pattern as `bolt402-lnd`:

- Implements `LnBackend` trait from `bolt402-core`
- Uses CLN's gRPC interface via `tonic` with vendored proto files
- Supports mTLS authentication (client certificate + CA certificate)
- Provides `ClnBackend::connect()` and `ClnBackend::from_env()` constructors

### Architecture

```
bolt402-core (LnBackend trait)
      ↑
bolt402-cln (ClnBackend struct)
      ↑
CLN gRPC API (node.proto + primitives.proto)
```

### CLN gRPC API Mapping

| `LnBackend` method | CLN gRPC RPC | Notes |
|---|---|---|
| `pay_invoice()` | `Pay(PayRequest)` | Returns `PayResponse` with preimage, amounts |
| `get_balance()` | `ListFunds(ListfundsRequest)` | Sum `our_amount_msat` across channels |
| `get_info()` | `Getinfo(GetinfoRequest)` | Returns node id, alias, active channels |

### Key Design Decisions

1. **`Pay` RPC over `SendPay`**: CLN's `Pay` is the high-level payment command (handles pathfinding, retries). `SendPay` is low-level and requires manual route construction. `Pay` is the right choice for an SDK.

2. **Vendored proto files**: Like `bolt402-lnd`, we vendor `node.proto` and `primitives.proto` from the CLN repository. This avoids depending on the `cln-grpc` crate (which pulls in the entire CLN dependency tree) and gives us control over proto compatibility.

3. **mTLS authentication**: CLN's gRPC uses mutual TLS. The client needs:
   - CA certificate (`ca.pem`) — to verify the server
   - Client certificate (`client.pem`) — to authenticate the client
   - Client key (`client-key.pem`) — private key for the client cert
   These are generated by CLN in `~/.lightning/<network>/`.

4. **Max fee via `maxfee`**: CLN's `Pay` accepts `maxfee` (absolute max fee in msat) rather than LND's percentage-based fee limit. We convert the `max_fee_sats` parameter to millisatoshis and pass it as `maxfee`.

5. **Balance from `ListFunds`**: CLN has no direct "channel balance" RPC. We use `ListFunds` and sum `our_amount_msat` across all channels in normal state.

6. **Node ID bytes to hex**: CLN returns node IDs as raw bytes in gRPC. We hex-encode them for `NodeInfo.pubkey`.

### API Sketch

```rust
use bolt402_cln::ClnBackend;

// Direct connection
let backend = ClnBackend::connect(
    "https://localhost:9736",
    "/path/to/ca.pem",
    "/path/to/client.pem",
    "/path/to/client-key.pem",
).await?;

// From environment variables
let backend = ClnBackend::from_env().await?;
// Reads: CLN_GRPC_HOST, CLN_CA_CERT_PATH, CLN_CLIENT_CERT_PATH, CLN_CLIENT_KEY_PATH
```

### Error Types

```rust
pub enum ClnError {
    Transport(tonic::transport::Error),
    Rpc(tonic::Status),
    Io(std::io::Error),
    Payment(String),
}
```

Maps to `ClientError` via `From<ClnError>`:
- `Payment` → `ClientError::PaymentFailed`
- Others → `ClientError::Backend`

### Alternatives Considered

1. **Use `cln-grpc` crate directly**: Rejected. The `cln-grpc` crate brings in heavy CLN dependencies. Vendoring the two proto files (~50KB + ~3KB) is simpler, lighter, and consistent with how we handle LND.

2. **JSON-RPC over Unix socket**: CLN also supports JSON-RPC. Rejected because gRPC is the recommended remote interface, supports mTLS natively, and aligns with our LND approach.

3. **Use `cln-rpc` crate (JSON-RPC)**: Rejected for the same reasons. gRPC is more robust for remote connections.

### Testing Plan

- Unit tests: error type conversions, config validation, debug format (no secret leaks)
- `from_env()` tests: missing env vars, invalid values
- Invalid connection tests: bad URIs, bad cert paths
- Integration testing: would require a live CLN node (out of scope for CI, can test manually)
