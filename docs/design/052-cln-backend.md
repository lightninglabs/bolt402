# Design: CLN (Core Lightning) Backend Adapter

**Issue:** #52
**Author:** Dario Anongba Varela
**Date:** 2026-03-21
**Status:** Implemented

## Problem

L402sdk currently supports LND, NWC, and SwissKnife as Lightning backends. Core Lightning (CLN) is the second most widely deployed Lightning implementation, maintained by Blockstream. Users running CLN nodes need first-class support for both CLN's native gRPC interface and the REST API used in browser and edge environments.

## Proposed Design

### Crate: `l402-cln`

Follows the same adapter pattern as `l402-lnd`:

- Implements `LnBackend` trait from `l402-proto`
- Uses CLN's gRPC interface via `tonic` with vendored proto files
- Uses CLN REST API with rune authentication
- Supports mTLS authentication for gRPC and rune authentication for REST
- Provides `ClnGrpcBackend::connect()`, `ClnGrpcBackend::from_env()`, `ClnRestBackend::new()`

### Architecture

```
l402-proto (LnBackend trait)
      ↑
l402-cln
  ├── ClnGrpcBackend ──► CLN gRPC API
  └── ClnRestBackend ──► CLN REST API
```

### CLN API Mapping

| `LnBackend` method | CLN gRPC RPC | Notes |
|---|---|---|
| `pay_invoice()` | `Pay(PayRequest)` | Returns `PayResponse` with preimage, amounts |
| `get_balance()` | `ListFunds(ListfundsRequest)` | Sum `our_amount_msat` across channels |
| `get_info()` | `Getinfo(GetinfoRequest)` | Returns node id, alias, active channels |

The REST backend maps to the same CLN operations through REST endpoints:

| `LnBackend` method | CLN REST endpoint | Notes |
|---|---|---|
| `pay_invoice()` | `POST /v1/pay` | Sends `bolt11` and fee limits |
| `get_balance()` | `POST /v1/listfunds` | Sums spendable channel balances |
| `get_info()` | `POST /v1/getinfo` | Returns node metadata |

### Key Design Decisions

1. **`Pay` RPC over `SendPay`**: CLN's `Pay` is the high-level payment command (handles pathfinding, retries). `SendPay` is low-level and requires manual route construction. `Pay` is the right choice for an SDK.

2. **Vendored proto files**: Like `l402-lnd`, we vendor `node.proto` and `primitives.proto` from the CLN repository. This avoids depending on the `cln-grpc` crate (which pulls in the entire CLN dependency tree) and gives us control over proto compatibility.

3. **mTLS authentication (gRPC)**: CLN's gRPC uses mutual TLS. The client needs:
   - CA certificate (`ca.pem`) — to verify the server
   - Client certificate (`client.pem`) — to authenticate the client
   - Client key (`client-key.pem`) — private key for the client cert
   These are generated by CLN in `~/.lightning/<network>/`.

4. **Rune authentication (REST)**: CLN REST uses runes for authentication. Runes are CLN's native bearer token system, sent via the `Rune` HTTP header.

5. **Max fee via `maxfee`**: CLN's `Pay` accepts `maxfee` (absolute max fee in msat) rather than LND's percentage-based fee limit. We convert the `max_fee_sats` parameter to millisatoshis and pass it as `maxfee`.

6. **Balance from `ListFunds`**: CLN has no direct "channel balance" RPC. We use `ListFunds` and sum `our_amount_msat` across all channels in normal state.

7. **Node ID bytes to hex**: CLN returns node IDs as raw bytes in gRPC. We hex-encode them for `NodeInfo.pubkey`.

### API Sketch

```rust
use l402_cln::{ClnGrpcBackend, ClnRestBackend};

// gRPC with mTLS
let grpc = ClnGrpcBackend::connect(
    "https://localhost:9736",
    "/path/to/ca.pem",
    "/path/to/client.pem",
    "/path/to/client-key.pem",
).await?;

// REST with rune auth
let rest = ClnRestBackend::new(
    "https://localhost:3010",
    "rune-token-value...",
)?;
```

### Error Types

```rust
pub enum ClnError {
    Transport(tonic::transport::Error),
    Rpc(tonic::Status),
    Io(std::io::Error),
    Payment(String),
}
```

Maps to `ClientError` via `From<ClnError>`:
- `Payment` → `ClientError::PaymentFailed`
- Others → `ClientError::Backend`

### Alternatives Considered

1. **Use `cln-grpc` crate directly**: Rejected. The `cln-grpc` crate brings in heavy CLN dependencies. Vendoring the two proto files (~50KB + ~3KB) is simpler, lighter, and consistent with how we handle LND.

2. **JSON-RPC over Unix socket**: CLN also supports JSON-RPC. Rejected because gRPC is the recommended remote interface, supports mTLS natively, and aligns with our LND approach.

3. **Use `cln-rpc` crate (JSON-RPC)**: Rejected for the same reasons. gRPC is more robust for remote connections.

### Testing Plan

- Unit tests: error type conversions, config validation, debug format (no secret leaks)
- `from_env()` tests: missing env vars, invalid values
- Invalid connection tests: bad URIs, bad cert paths
- Regtest integration testing in `tests/regtest/` covering CLN gRPC and CLN REST through Aperture

```
├── Cargo.toml
├── build.rs           # tonic-build for CLN protos
├── proto/
│   ├── node.proto     # Vendored from ElementsProject/lightning
│   └── primitives.proto
└── src/
    ├── lib.rs         # Re-exports, generated module includes
    ├── grpc.rs        # ClnGrpcBackend implementation
    ├── rest.rs        # ClnRestBackend implementation
    └── error.rs       # ClnError type
```
