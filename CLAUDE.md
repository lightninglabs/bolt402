# CLAUDE.md — Instructions for AI Coding Agents

## Project

bolt402 — L402 client SDK for AI agent frameworks. Rust workspace.

## Quick Start

```bash
cargo build          # Build all crates
cargo test           # Run all tests
cargo fmt --check    # Check formatting
cargo clippy         # Lint
cargo doc --no-deps  # Build docs
```

## Architecture

Hexagonal / Clean Architecture. Read `AGENTS.md` for the full breakdown.

- `bolt402-proto`: Protocol types (L402Challenge, L402Token, L402Error). No internal deps.
- `bolt402-core`: Client engine, ports (LnBackend, TokenStore traits), adapters (InMemoryTokenStore, BudgetTracker, Receipt).
- `bolt402-lnd`: LND gRPC adapter (implements LnBackend).
- `bolt402-mock`: Mock L402 server for testing.

## Rules

1. **No TODOs or placeholders.** Implement fully or don't implement.
2. **Every public item must have documentation.** `#![warn(missing_docs)]` is enforced.
3. **Tests for every module.** Unit tests inline, integration tests in `tests/`.
4. **Conventional commits:** `feat:`, `fix:`, `chore:`, `test:`, `docs:`, `ci:`, `refactor:`
5. **One PR per logical change.** Squash merge. Clean history.
6. **No AI watermarks.** No `Co-authored-by: Claude`, no `Generated by AI`, nothing.
7. **Git config:** `Dario Anongba Varela <dario.anongba@gmail.com>`
8. **Error handling:** `thiserror` for typed errors. No `unwrap()` in library code. No `anyhow` in library crates.
9. **Async:** All port traits use `async_trait`. Adapters use `tokio`.
10. **Format:** Run `cargo fmt` before committing. Run `cargo clippy` and fix all warnings.

## Dependencies (workspace-level)

Check `Cargo.toml` at workspace root for pinned versions. Don't add new dependencies without justification.

## Style Guide

- Prefer `impl` blocks over free functions
- Use `Self` in impl blocks where possible
- Group imports: std, external crates, internal crates
- Error variants should be descriptive (include context in the message)
- Prefer `&str` over `String` in function parameters where ownership isn't needed

## What NOT to Do

- Don't add `println!` for debugging — use `tracing`
- Don't use `unsafe` without extremely good reason
- Don't add feature flags unless architecturally necessary
- Don't break the public API without discussion
- Don't commit generated files (target/, *.lock for libraries)
